採用アーキテクチャ：レイヤードアーキテクチャ

参考記事内 で紹介されているのは レイヤードアーキテクチャ をベースに
いろいろカスタマイズされたものらしいです。

クリーンアーキテクチャに似たアーキテクチャだとか。

ユースケース層という呼び方はクリーンアーキテクチャ由来ですね。

DDD の文脈だと アプリケーション層 と呼ばれますが、
アプリケーションって意味が広くて分かりづらいので、
本プロジェクトでは ユースケース という単語を使用します。

とりあえず、今回はスライドページ19で紹介されているディレクトリ構成に従って、
DDD を意識して Web API を実装していこうと思います。

（意識だけして、実践できずに終わりましたが😇）


レイヤードアーキテクチャ における各層の依存関係 について説明します。

依存関係の図は下記のとおりです。

矢印は依存の方向を示しています。
例えば、上図だと Handler層 は UseCase層 の処理を利用することを意味します。


一般的なレイヤードアーキテクチャだと、上から下に一方向に依存します。
しかし、今回は、Infra層が Domain層に依存しています。
このあたりはオニオンアーキテクチャーやクリーンアーキテクチャと同じやり方ですね。

さきほどの図を視点を変えて見てみます。（下記の図）
今回採用したアーキテクチャは、
オニオンアーキテクチャーやクリーンアーキテクチャのように、
依存が中心方向に のみ 向いていることがわかります。

すべての依存が中心に向かっているこの状態が理想です。

（Handler層と Infra層が一緒の層みたいになっていますが、全くの別物です。うまく分離して描けず、こうなりました。ご注意を）

依存関係について、もう少し述べておくと、
基本的に依存はひとつ下の層までに抑えておくべきのようです。

ただし、簡略化のために2つ下の層まで依存している例もあるので、
そこはチームとして同意が取れていれば良いのではないでしょうか。


ここで、ユーザから APIリクエスト があった場合を考えてみます。

ユーザからのリクエストは Handler で受け取られ、 UseCase を使って処理が行われます。
さらに、UseCase は Domain を使って処理を行います。
ここまでは処理が中心に進んでいる、つまり依存は中心に向かって発生しています。

しかし、たいていのサービスって DB を使用しますよね。
つまり、ユースケースからドメインを介して、Infra を利用することになります。

UseCase → Domain → Infra

…依存が外側を向いてしまいました。
これは許されていません。ではどうするか。

依存性逆転の法則 を使います。


依存性逆転の法則 とは、 interfaceを利用して、依存の方向を逆にすること です。

もう少し詳しく説明します。
まず、 ① Domain層 において、 DB とのやりとりを interface で定義しておきます。
interface （後ほどコード内にて BookRepository として出てきます） 自体は実装を持たないので、
どこにも依存していません。

次に、 ② Infra層 から Domain層 に定義した interface （後ほどコード内にて BookPersistence として出てきます） を実装します。


①, ② の2ステップを踏むことで、まず Domain は interface に対して 処理をお願いするだけでよくなります。 先ほども言ったとおり interface は 実装を持たないので依存関係はありません。

interface 自体は実装を持ちませんが、
Infra が interface の実装を行っているので、ちゃんとDBアクセスして処理を行うことができます。

ここで、 Infra は interface を実装しているので、依存が interface 、すなわち Domain に向いています。

依存性が逆転し、すべての依存関係が中心に向かうようになりましたね。


ここはとっつきづらいところなので、まだいまいち理解できないかもしれません。

以降、実際のコードを紹介していくので、コードに落とし込みながら考えてみてください。

書籍一覧を取得するAPIを作る

Domain 層

まずは、/domain/model に書籍モデルを作っていきます。

Domain層 はシステムが扱う業務領域に関するコードを置くところです。

よって、「書籍」 がどういうものなのかモデルという形で定義します。

/domain/model/book.go

 1package model
 2
 3import "time"
 4
 5// Book : Book を表すドメインモデル
 6// !! 重度のドメイン貧血症です !!
 7type Book struct {
 8	Id       int64
 9	Title    string
10	Author   string
11	IssuedAt time.Time
12}

冒頭でも述べたとおり、みごとなドメイン貧血症っぷりです。

ちゃんと 値オブジェクトを使ったりして、ごりごり DDD していきたいですが、今回は…省きます🙇‍♂️


次に、/domain/repository/book.go を作っていきます。

今回、 リポジトリでやることを簡単に言うと、 DB や KVS などで行う CRUD処理 の定義です。
ただし、Domain層には技術的関心事を持ち込まない というルールがあるため、
ここでは interface を定義するだけです。

実装は、後述する infra で行います。
（Infra層 は技術的関心事を扱う層です）

リポジトリについてちゃんと知りたい方は、
こちらが参考になると思います。


/domain/repository/book.go

 1package repository
 2
 3import (
 4	"context"
 5
 6	"github.com/yyh-gl/go-api-server-by-ddd/domain/model"
 7)
 8
 9// BookRepository : Book における Repository のインターフェース
10//  -> 依存性逆転の法則により infra 層は domain 層（本インターフェース）に依存
11type BookRepository interface {
12	GetAll(context.Context) ([]*model.Book, error)
13}

今は 全ての書籍を取得する関数 GetAll() のみ定義します。


ここで、はじめに示した 依存関係の図 を思い出してください。

今定義した Domain層 は他の層のコードを一切利用していません。
つまり、下図の赤枠の中で依存関係が完結しています。


Infra 層

さきほど述べたとおり、Infra層 は技術的関心事を扱う層です。

ここでさっき定義した repository の処理を実装します。

/infra/persistence/book.go

 1package persistence
 2
 3// repository という名前にしたいが domain 配下の repository とパッケージ名が被ってしまうため persistence で代替
 4
 5import (
 6	"context"
 7	"time"
 8
 9	"github.com/yyh-gl/go-api-server-by-ddd/domain/model"
10	"github.com/yyh-gl/go-api-server-by-ddd/domain/repository"
11)
12
13type bookPersistence struct{}
14
15// NewBookPersistence : Book データに関する Persistence を生成
16func NewBookPersistence() repository.BookRepository {
17	return &bookPersistence{}
18}
19
20// GetAll : DB から Book データを全件取得（BookRepository インターフェースの GetAll() を実装したもの）
21//  -> 本来は DB からデータを取得するが、簡略化のために省略（モックデータを返却）
22func (bp bookPersistence) GetAll(context.Context) ([]*model.Book, error) {
23	book1 := model.Book{}
24	book1.Id = 1
25	book1.Title = "DDDが分かる本"
26	book1.Author = "たろうくん"
27	book1.IssuedAt = time.Now().Add(-24 * time.Hour)
28
29	book2 := model.Book{}
30	book2.Id = 2
31	book2.Title = "レイヤードアーキテクチャが分かる本"
32	book2.Author = "はなこさん"
33	book2.IssuedAt = time.Now().Add(-24 * 7 * time.Hour)
34
35	return []*model.Book{&book1, &book2}, nil
36}

なお、 実際には DB にアクセスし、データを持ってくるようにします。
ここでは一旦モックデータを返すようにしておきます。

また、Persistence という単語がいきなり出てきましたが、これは Repository と同義です。
実際に NewBookPersistence() の中身を見ると Repository のインターフェースを返していると思います。
（NewBookPersistence()の詳細は後述）

本当は Repositoryという名前を使いたかったのですが、
Domain層と Infra層 でパッケージ名が被ってしまうため、やむなくこうしています。


先ほどと同様に 依存関係 を確認します。

Infra層 は Domain層 で作った /domain/repository/book.go のインターフェース（BookRepository）を実装しています。


ここで、Golang に慣れていない方は、どこでインターフェースと関連づけてるの？
という疑問が生まれると思います。

答えは、 NewBookPersistence() です。
この関数の戻り値は インターフェース です。
したがって、17行目でreturnする bookPersistence がインターフェースを満たしていないとエラーとなります。
このようにして インターフェースを満たしているか否かを判別します。

NewBookPersistence() をどこで使うかは後述します。


では、依存関係を見ていきます。
上述したとおり、Infra層 は Domain層 のインターフェースを満たすように作られているので、Domain層に依存しています。
Golang には implements とかないので分かりづらいですね。
でも、確かに依存しています。


UseCase 層

UseCase層 では、システムのユースケースを満たす処理の流れを実装します。

今回は単純な処理しかしないので、この層の存在価値が少し分かりづらくなってしまいます。

複雑なビジネスロジックがあるときは、この層の存在が効いてくると思います。


コードは以下のとおりです。

/usecase/book.go

 1package usecase
 2
 3import (
 4	"context"
 5
 6	"github.com/yyh-gl/go-api-server-by-ddd/domain/model"
 7	"github.com/yyh-gl/go-api-server-by-ddd/domain/repository"
 8)
 9
10// BookUseCase : Book における UseCase のインターフェース
11type BookUseCase interface {
12	GetAll(context.Context) ([]*model.Book, error)
13}
14
15type bookUseCase struct {
16	bookRepository repository.BookRepository
17}
18
19// NewBookUseCase : Book データに関する UseCase を生成
20func NewBookUseCase(br repository.BookRepository) BookUseCase {
21	return &bookUseCase{
22		bookRepository: br,
23	}
24}
25
26// GetAll : Book データを全件取得するためのユースケース
27//  -> 本システムではあまりユースケース層の恩恵を受けれないが、もう少し大きなシステムになってくると、
28//    「ドメインモデルの調節者」としての役割が見えてくる
29func (bu bookUseCase) GetAll(ctx context.Context) (books []*model.Book, err error) {
30	// Persistence（Repository）を呼出
31	books, err = bu.bookRepository.GetAll(ctx)
32	if err != nil {
33		return nil, err
34	}
35	return books, nil
36}

UseCase層 の依存関係も見てみましょう。

UseCase層 は /domain/repository を呼び出しています。
したがって、 UseCase層 は Domain層 に依存しています。


参考にしている資料 では、
UseCase層 をさらに input と output で切っていますが、複雑になりすぎると思い、省略しました。


Handler 層

次に Handler層 です。

本プロジェクトにおける、Handler層 の役目は、HTTPリクエストを受け取り、UseCase を使って処理を行い、結果を返す ことです。

ただし、本来の Handler層は HTTPリクエストに限った話ではありません。

外部にあるものがなんであれ、その差異を吸収して、ユースケースに伝えるのが役目です。

したがって、HTTP通信以外でも対応できるように、本プロジェクトでは /handler/rest というふうにディレクトリを切っています。
（RESTful API であることを明確にしてみました）
CLIを追加するなら /handler/cli というふうにディレクトリを切ればOK。


本プロジェクトのコード的には以下のようになります。

/handler/blog.go

 1package rest
 2
 3// Handler 層を変えるだけで、例えば CLI にも簡単に対応可能
 4
 5import (
 6	"encoding/json"
 7	"net/http"
 8	"time"
 9
10	"github.com/julienschmidt/httprouter"
11	"github.com/yyh-gl/go-api-server-by-ddd/usecase"
12)
13
14// BookHandler : Book における Handler のインターフェース
15type BookHandler interface {
16	Index(http.ResponseWriter, *http.Request, httprouter.Params)
17}
18
19type bookHandler struct {
20	bookUseCase usecase.BookUseCase
21}
22
23// NewBookUseCase : Book データに関する Handler を生成
24func NewBookHandler(bu usecase.BookUseCase) BookHandler {
25	return &bookHandler{
26		bookUseCase: bu,
27	}
28}
29
30// BookIndex : GET /books -> Book データ一覧を返す
31func (bh bookHandler) Index(w http.ResponseWriter, r *http.Request, pr httprouter.Params) {
32	// request : 本 API のリクエストパラメータ
33	//  -> こんな感じでリクエストも受け取れますが、今回は使いません
34	type request struct {
35		Begin uint `query:"begin"`
36		Limit uint `query:"limit"`
37	}
38
39	// bookField : response 内で使用する Book を表す構造体
40	//  -> ドメインモデルの Book に HTTP の関心事である JSON タグを付与したくないために Handler 層で用意
41	//     簡略化のために JSON タグを付与したドメインモデルを流用するプロジェクトもしばしば見かける
42	type bookField struct {
43		Id       int64     `json:"id"`
44		Title    string    `json:"title"`
45		Author   string    `json:"author"`
46		IssuedAt time.Time `json:"issued_at"`
47	}
48
49	// response : 本 API のレスポンス
50	type response struct {
51		Books []bookField `json:"books"`
52	}
53
54	ctx := r.Context()
55
56	// ユースケースの呼出
57	books, err := bh.bookUseCase.GetAll(ctx)
58	if err != nil {
59		// TODO: エラーハンドリングをきちんとする
60		http.Error(w, "Internal Server Error", 500)
61		return
62	}
63
64	// 取得したドメインモデルを response に変換
65	res := new(response)
66	for _, book := range books {
67		var bf bookField
68		bf = bookField(*book)
69		res.Books = append(res.Books, bf)
70	}
71
72	// クライアントにレスポンスを返却
73	w.Header().Set("Content-Type", "application/json")
74	if err = json.NewEncoder(w).Encode(res); err != nil {
75		// TODO: エラーハンドリングをきちんとする
76		http.Error(w, "Internal Server Error", 500)
77		return
78	}
79}

依存関係は以下のとおりです。

57行目で UseCase を使用するので、UseCase層に依存しています。


main.go

ここまでで、書籍に関する Handler, UseCase, Repository が用意できました。
最後に、main.go にて、これらの依存関係を定義してやることで、利用可能な状態にします。
（DI とかはやってません🙏）

このとき利用するのが、各層に用意されている NewXxx() という関数です。

NewXxx() を使用して、Handler や UseCase, Repository を生成し、メソッドを実行できるようにします。

/cmd/api/main.go

 1package main
 2
 3import (
 4	"fmt"
 5	"log"
 6	"net/http"
 7
 8	"github.com/julienschmidt/httprouter"
 9	handler "github.com/yyh-gl/go-api-server-by-ddd/handler/rest"
10	"github.com/yyh-gl/go-api-server-by-ddd/infra/persistence"
11	"github.com/yyh-gl/go-api-server-by-ddd/usecase"
12)
13
14func main() {
15	// 依存関係を注入（DI まではいきませんが一応注入っぽいことをしてる）
16	// DI ライブラリを使えば、もっとスマートになるはず
17	bookPersistence := persistence.NewBookPersistence()
18	bookUseCase := usecase.NewBookUseCase(bookPersistence)
19	bookHandler := handler.NewBookHandler(bookUseCase)
20
21	// ルーティングの設定
22	router := httprouter.New()
23	router.GET("/api/v1/books", bookHandler.Index)
24
25	// サーバ起動
26	fmt.Println("========================")
27	fmt.Println("Server Start >> http://localhost:3000")
28	fmt.Println("========================")
29	log.Fatal(http.ListenAndServe(":3000", router))
30}

注目していただきたのが、17行目から19行目の処理です。
ここで、各層の NewXxx() の処理を使って依存関係を定義しています。

DI ライブラリを使うことで、よりスマートに書けると思いますが、
愚直にやるならこんな感じです。
